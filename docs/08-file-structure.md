# File Structure

Where things live inside a TLB internal app. Predictable structure is the cheapest tool we have for keeping AI agents and humans aligned.

## Top-Level Layout

The canonical structure for a Vite plus React plus TypeScript TLB internal app:

```
repo-root/
├── AGENTS.md
├── CLAUDE.md
├── .cursorrules
├── README.md
├── components.json              shadcn CLI config
├── eslint.config.js
├── package.json
├── tsconfig.json
├── vite.config.ts
└── src/
    ├── main.tsx                 Entry point
    ├── App.tsx                  Top-level providers, router, Toaster
    ├── routes/                  Or pages/. Route components only.
    ├── components/
    │   ├── ui/                  shadcn primitives (generated, editable)
    │   ├── layout/              AppShell, Sidebar, TopBar, PageHeader, ThemeToggle
    │   ├── forms/               FormField wrappers, form-specific composites
    │   ├── data/                DataTable, EmptyState, KpiTile, ChartCard, ErrorState
    │   ├── feedback/            ConfirmDialog, Toast wrappers
    │   ├── auth/                Permission, RoleGate, SignInButton
    │   ├── icons/               Custom domain icons (rare)
    │   └── feature/             Feature-specific composites
    │       └── <feature>/
    │           ├── <feature>-table.tsx
    │           ├── <feature>-form.tsx
    │           └── <feature>-detail-sheet.tsx
    ├── hooks/                   Custom React hooks, one per file
    ├── lib/
    │   ├── utils.ts             cn() and any pure helpers
    │   ├── api.ts               Fetch client config
    │   ├── auth.ts              MSAL helpers
    │   └── format.ts            Intl wrappers for currency, date, number
    ├── schemas/                 zod schemas, one per resource
    │   ├── vendor.ts
    │   └── invoice.ts
    ├── stores/                  zustand stores. Only when needed.
    ├── types/                   Shared types and library type augmentations
    ├── styles/
    │   └── tailwind.css         Tailwind v4 entry + @theme tokens
    └── assets/
        └── logos/               TLB logo SVG variants
```

For monorepos (like both current repos), the same structure lives inside the client package:

```
repo-root/
├── packages/
│   ├── client/                  Same structure as above
│   ├── shared/                  Cross-package types, schemas, constants
│   └── api/                     Backend (NestJS, Azure Functions, etc.)
```

## File Naming

| Kind | Convention | Example |
|---|---|---|
| React component file | `kebab-case.tsx` | `vendor-form.tsx` |
| React component export | `PascalCase` | `export function VendorForm()` |
| Hook file | `kebab-case.ts` with `use-` prefix | `use-debounced-value.ts` |
| Hook export | `camelCase` with `use` prefix | `useDebouncedValue` |
| Schema file | `kebab-case.ts` matching the resource | `vendor.ts` |
| Schema export | `camelCase` plus `Schema` | `vendorSchema` |
| Type-only file | `kebab-case.ts` | `vendor.types.ts` |
| Test file | colocated, `.test.ts` or `.test.tsx` | `vendor-form.test.tsx` |
| Constants file | `kebab-case.ts` | `payment-terms.ts` |

Rules:
- One component per file. No multi-component files. Helper components used only by one parent live in the same file at the bottom or as a private sub-component.
- File names match the primary export. If a component is `VendorForm`, the file is `vendor-form.tsx`.

## Import Conventions

All cross-folder imports use the `@/` alias. Relative imports are reserved for adjacent files.

```ts
// Good
import { Button } from "@/components/ui/button";
import { vendorSchema } from "@/schemas/vendor";
import { cn } from "@/lib/utils";

// Good (sibling file)
import { VendorFormFields } from "./vendor-form-fields";

// Bad
import { Button } from "../../components/ui/button";
```

Import order, top to bottom:

1. React and React DOM
2. Third-party libraries
3. shadcn UI primitives (`@/components/ui/*`)
4. TLB composites (`@/components/*`)
5. Hooks (`@/hooks/*`)
6. Lib and schemas (`@/lib/*`, `@/schemas/*`)
7. Types (`@/types/*`)
8. Relative imports
9. CSS imports

Blank line between groups. ESLint can enforce this; we leave it as a guideline rather than a strict lint to keep configuration light.

## What Lives Where (Decision Aid)

When you have a new component, ask in order:

- Is it a primitive from shadcn? It lives in `src/components/ui/`. (Generated by the CLI; do not put hand-rolled files here.)
- Is it part of the app shell (sidebar, top bar, header)? `src/components/layout/`.
- Does it appear in two or more features and present data? `src/components/data/`.
- Is it form-related plumbing? `src/components/forms/`.
- Is it specific to one feature? `src/components/feature/<feature>/`.

When in doubt, scope it to the feature folder first. Promote to a shared folder only when reused.

## What Should Not Be in `src/`

- Mock data. Lives in `src/__fixtures__/` only for tests. Production data ships from the API.
- Environment-specific config. Use `.env` files and Vite's `import.meta.env`.
- TODO files, scratch notes, draft components. Use a branch and PR description instead.
- Auto-generated files outside of `src/components/ui/`. If a generator is used (OpenAPI types, GraphQL codegen), generated output lives in `src/generated/` and is gitignored.

## Two-Repo Reality

Both `tlb-financial-tools` and `improv-procurement` already deviate from this structure in places. The audits in `repo-audits/` document the gaps. The structure above is the target. Migrations move files in their natural follow-up PR cycle, not as a single sweeping rename.
